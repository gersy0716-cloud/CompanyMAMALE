# 项目开发流程规范

> 核心理念：从"命令式编码"转变为"技术合伙人对话"，每个阶段都需要缜密推进。

---

## 📋 四阶段流程

```
┌────────────────────────────────────────────────────────────────────────────┐
│  1️⃣ 需求讨论  →  2️⃣ 方案规划  →  3️⃣ 领导审核  →  4️⃣ 代码实现           │
│  (P-角色扮演)    (P-风险预警)    (用户确认)     (R1-编码 + R2-审查)       │
└────────────────────────────────────────────────────────────────────────────┘
```

> [!CAUTION]
> **严禁跳过阶段1-3直接编码！** 每个新项目必须经过完整流程。

---

## 阶段1️⃣：需求讨论 💬

与用户充分讨论项目需求：

- 明确功能范围、UI设计风格、技术栈
- 讨论数据库设计思路
- 询问所有不确定的细节

**P-R-R 策略应用**：

| 策略 | 执行方式 | 目的 |
|:---|:---|:---|
| **角色扮演** | 扮演资深架构师，从安全性、可扩展性、性能角度分析 | 输出全面设计考量 |
| **探索性提问** | 比较多种技术方案的优缺点 | 评估最优路径 |
| **风险预警** | 扮演"黑客/测试工程师"，列出常见人为错误和安全漏洞 | 提前发现潜在坑点 |

---

### 核心工具：PRP (Context Engineering)

- **输入准备**：利用 [docs/INITIAL.md](../docs/INITIAL.md) 梳理需求。
- **蓝图输出**：生成一份详尽的 `PRP.md`（Product Requirements Prompt），包含全局规则、代码规范与示例。
- **目的**：确保 AI 彻底理解"标准答案长什么样"。

将文档保存到项目文件夹中。

---

## 阶段3️⃣：领导审核 ✅

- 使用 `notify_user` 请求用户审核实施方案
- 使用 `notify_user` 请求用户审核数据库表设计
- 等待用户确认方案无误
- 根据用户反馈调整方案

---

## 阶段4️⃣：代码实现 💻

在用户审核通过后，才开始编写实际代码。

### 核心工具：Planning with Files (Session Memory)

- **三文件记录**：必须同步维护 `task_plan.md`、`findings.md` 和 `progress.md`。
- **目的**：将"工作记忆"持久化到磁盘，防止在高频对话中发生"目标漂移"或因 `/clear` 导致的记忆丧失。

### R1-请求与实现策略

| 策略 | 执行方式 | 目的 |
|:---|:---|:---|
| **验收标准** | 使用 Given-When-Then 格式定义行为 | 消除需求歧义 |
| **错误处理** | 为所有失败情况实现 try-catch | 覆盖失败路径 |
| **约束声明** | 遵循项目规范（类型定义、ESLint、命名规范） | 保证代码质量底线 |

**验收标准示例**：

```gherkin
- Given: 密码字符串
- When: 字符串长度小于8位
- Then: 返回 { valid: false, error: '长度不能小于8位' }
```

### R2-审查与重构策略（代码完成后）

| 策略 | 执行方式 | 目的 |
|:---|:---|:---|
| **生成测试** | 编写单元测试，包含边缘情况 | 检验逻辑缜密性 |
| **优化审查** | 审视性能、可读性、是否有更现代的写法 | 追求更优解 |
| **安全审计** | 扮演安全专家，检查SQL注入、XSS | 堵住安全漏洞 |

---

## 重要原则

- **不要提前编码**：在实施方案确认前，不要创建代码文件
- **严格遵循流程**：每个阶段都需要用户的明确许可才能进入下一阶段
- **高风险操作手动执行**：AI 助手不应自动运行涉及移动大量文件或重组目录的指令。必须提供代码清单由用户手动执行。

---

## 🧠 记忆维护规范（强制）

> [!CAUTION]
> 以下规则确保记忆系统不退化。违反这些规则会导致 Memory 文件膨胀为大杂烩。

### 三层上下文路由表

在记录新信息时，**必须**根据内容性质选择正确的存放位置：

| 信息类型 | 存放位置 | 示例 |
| :--- | :--- | :--- |
| AI 必须遵守的规范 | `.claude/rules/` | 编码标准、开发流程、目录结构 |
| UI/审美偏好 (动态积累) | `memory/UI设计.md` | 喜欢的配色、圆角大小、玻璃拟态等 |
| 技术栈/架构/项目结构 | `docs/PLANNING.md` | 用什么框架、目录怎么组织 |
| API/库/工具文档 | `.claude/skills/` | 数据库 API 怎么用 |
| 踩过的坑、Bug 教训 | `memory/` | CORS 问题、各种陷阱 |

### Memory 文件写入规则

1. **只写踩坑**：Memory 只记录"踩过的坑"和"学到的教训"，不写规范、不写架构、不写偏好
2. **归类不堆积**：新踩坑必须归入已有分类（环境/代码/数据库等），**禁止**在文件底部按日期追加流水帐
3. **精简表达**：问题→原因→解决方案，三行搞定，不写长篇大论
4. **不重复 rules**：如果内容已经在 `rules/` 中存在，不要在 Memory 中重复
5. **定期清理**：已修复且不会再犯的坑，可以删除

### 反模式（禁止）

```
❌ 在 rules/ 中写动态变化的 UI 审美（应在 memory/）
❌ 在 memory/ 中写技术栈列表（应在 docs/PLANNING.md）
❌ 在 memory/ 中写开发流程（应在 rules/development-flow.md）
❌ 在 memory/ 底部按日期追加日志
❌ 同一条信息出现在两个以上文件中
```
